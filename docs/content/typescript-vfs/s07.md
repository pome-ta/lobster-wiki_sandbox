# 第7章 `createVirtualTypeScriptEnvironment()` のソースコードを読む（前編）

第6章では、

`createVirtualTypeScriptEnvironment()` が

> Virtual File System 上で動作する TypeScript の開発環境を構築する

ための関数であることを説明した。

しかし、これはまだ「役割」の説明に過ぎない。

本章からは、実際のソースコードを読みながら、

- 作者はどのような API を設計したのか
- なぜその引数が必要なのか
- 内部ではどのようなオブジェクトが生成されるのか

を順番に追っていく。

---

# 7.1 ソースコードを読む前に

OSS のソースコードを読むとき、

初心者ほど

```
関数本体
```

から読み始めてしまう。

しかし、

実際には

```
関数シグネチャ
```

だけでも、

かなり多くの情報を読み取ることができる。

そこで本章では、

実装を読む前に、

まず

```
公開 API
```

として設計された姿を見ていく。

---

# 7.2 関数シグネチャ

概念的には、

公開 API は次のようなシグネチャになっている。

```ts
export function createVirtualTypeScriptEnvironment(
    system,
    rootFiles,
    ts,
    compilerOptions
)
```

※実際の引数名や型はバージョンによって多少変化する可能性がある。

ここで重要なのは、

引数そのものより、

**この順番**

である。

作者は

```
System

↓

Root Files

↓

TypeScript

↓

Compiler Options
```

という順番で受け取るよう設計している。

つまり、

利用者は

まず

```
Virtual File System
```

を準備し、

その上で

```
どのファイルをコンパイル対象にするか
```

を指定し、

最後に

Compiler の設定を渡す、

という利用方法を想定していることが読み取れる。

---

## コラム：API は設計思想を表す

公開 API は、

単なる関数ではない。

ライブラリ作者が

> 利用者にどのように使ってほしいか

を表現したものである。

例えば、

もし

```ts
createVirtualTypeScriptEnvironment(
    compilerOptions,
    system
)
```

という順番だったら、

利用者は

まず設定を考え、

最後にファイルシステムを準備することになる。

しかし、

実際には逆である。

これは

**まず環境を作り、その上で Compiler を構成する**

という設計思想を表している。

---

# 7.3 第一引数 `system`

最初に渡されるのは

```
ts.System
```

である。

これは第4章で説明した

```
Compiler が利用するファイルシステム
```

そのものである。

ここで重要なのは、

Environment は

Virtual File System を作らない

ということである。

作成済みの

System

を受け取る。

つまり、

責務が完全に分離されている。

```
createSystem()

↓

System

↓

createVirtualTypeScriptEnvironment()
```

となる。

Environment が

System の生成まで担当してしまうと、

責務が大きくなり過ぎる。

作者は

この責務を分離している。

---

# 7.4 第二引数 `rootFiles`

次に渡されるのが

```
rootFiles
```

である。

これは

コンパイル対象となる

エントリーファイル

の一覧である。

例えば、

```text
/index.ts
```

だけをコンパイルする場合は、

```ts
[
    "/index.ts"
]
```

となる。

複数ファイルなら、

```ts
[
    "/index.ts",
    "/util.ts",
]
```

となる。

---

ここで重要なのは、

Compiler は

Map 全体をコンパイルするわけではない

という点である。

例えば、

Virtual File System に

```
1000ファイル
```

存在していても、

Root Files に

```
2ファイル
```

しか指定しなければ、

そこから import を辿って解析を開始する。

つまり、

Compiler が

「入口」

として扱うファイルを指定しているのである。

---

### コラム：なぜ `Map` 全体を解析しないのか

一見すると、

Map に存在するすべてのファイルを解析すれば良いように思える。

しかし、

それでは

```
サンプルコード

テストコード

古いコード

一時保存ファイル
```

まで解析対象になってしまう。

そのため、

Compiler は

入口となるファイルだけを受け取り、

そこから

```
import
```

を辿って

依存関係を構築する。

これは

通常の

```
tsc
```

と同じ動作である。

---

# 7.5 第三引数 `ts`

次に渡されるのが

```
ts
```

である。

ここで

初めて

TypeScript 本体が登場する。

これは

```ts
import ts from "typescript";
```

などで取得した

Compiler API

そのものである。

---

ここで疑問が生まれる。

なぜ

ライブラリ内部で

```
import "typescript"
```

しないのだろうか。

理由は、

利用者が使用している

TypeScript

のバージョンを尊重するためである。

例えば、

利用者が

```
TypeScript 5.9
```

を使用しているなら、

VFS 側も

同じインスタンスを利用する。

もし内部で別の TypeScript を読み込んでしまうと、

型や内部実装が一致しない可能性がある。

そのため、

TypeScript 本体は

利用者から渡してもらう設計になっている。

これは

Compiler API を扱うライブラリでは

よく採用される設計である。

---

# 7.6 第四引数 `compilerOptions`

最後に渡されるのが

Compiler Options

である。

これは

通常の

```
tsconfig.json
```

に相当する設定である。

例えば、

```ts
{
    target: ...,
    module: ...,
    strict: true,
}
```

などが含まれる。

ここで重要なのは、

Environment は

Compiler Options

を解釈しない

ということである。

解釈するのは

TypeScript Compiler

である。

Environment は

それを

Language Service

へ渡すだけである。

つまり、

ここでも責務は分離されている。

---

# 7.7 この関数から読み取れる設計思想

まだ関数本体は読んでいない。

しかし、

シグネチャだけで、

次のことが分かる。

- Virtual File System の生成は別関数。
- TypeScript 本体も別。
- Compiler 設定も別。
- この関数は「組み立て」に専念している。

つまり、

この関数は

**Builder**

に近い役割を持っている。

部品を作るのではなく、

部品を組み合わせる。

そのため、

処理の中心は

「生成」

ではなく

「接続」

になる。

---

## この章で理解できるようになったこと

- 関数シグネチャだけでも多くの設計情報を読み取れる。
- `createVirtualTypeScriptEnvironment()` は各部品を生成するのではなく受け取る。
- 引数の順番にも設計思想が表れている。
- この関数の責務は「部品を組み立てること」である。
