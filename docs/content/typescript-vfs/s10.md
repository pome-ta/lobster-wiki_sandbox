# 第5章 VirtualTypeScriptEnvironment（前編）

第4章では、`CompilerHost` の役割について説明した。

`CompilerHost` は、TypeScript Compiler がファイルへアクセスするための窓口であり、Virtual File System と Compiler を接続する Adapter の役割を担っている。

しかし、`CompilerHost` が完成しただけでは、エディタで利用される TypeScript の機能はまだ使用できない。

例えば、

- コード補完
- 型チェック
- 定義ジャンプ
- 参照検索
- リネーム
- シグネチャヘルプ

などの機能は、Compiler ではなく **Language Service** が提供している。

そのため、`@typescript/vfs` はさらに一段上のレイヤーとして、**VirtualTypeScriptEnvironment** という概念を提供している。

本章では、この VirtualTypeScriptEnvironment がどのような役割を持ち、なぜ必要なのかを理解していく。

---

# 5.1 VirtualTypeScriptEnvironment とは

まず最初に理解しておきたいのは、VirtualTypeScriptEnvironment は **TypeScript Compiler そのものではない** ということである。

また、**Language Service そのものでもない。** VirtualTypeScriptEnvironment は、Language Service を利用するために必要なオブジェクトを一つにまとめた「実行環境」である。

概念図で表すと、次のようになる。

```text
                 VirtualTypeScriptEnvironment
 ┌────────────────────────────────────────────┐
 │                                            │
 │  Virtual File System (Map)                 │
 │                                            │
 │  System                                    │
 │                                            │
 │  CompilerHost                              │
 │                                            │
 │  LanguageServiceHost                       │
 │                                            │
 │  LanguageService                           │
 │                                            │
 └────────────────────────────────────────────┘
```

Language Service 単体では、「現在どんなファイルが存在するのか」 「ファイルの内容は何か」 「編集されたか」 といった情報を知ることはできない。

そのため、Language Service が利用するための様々なオブジェクトをまとめたものが Environment である。

---

## Environment という名前の意味

ここで一つ疑問がある。

なぜ

```
VirtualLanguageService
```

ではなく、

```
VirtualTypeScriptEnvironment
```

という名前なのだろうか。

これは、返されるオブジェクトが Language Service 一つだけではないためである。

Environment は、Language Service を中心に、TypeScript を動作させるための実行環境全体を保持している。

つまり、利用者から見ると、

```
TypeScript が動作する仮想環境
```

そのものなのである。

そのため、Environment という名前が採用されている。

---

# 5.2 createVirtualTypeScriptEnvironment()

VirtualTypeScriptEnvironment を生成するために利用する公開 API が、`createVirtualTypeScriptEnvironment()` である。

ライブラリ利用者は、通常この関数を直接呼び出す。

概念的な利用例を示す。

```ts
const env = createVirtualTypeScriptEnvironment(
  system,
  rootFiles,
  ts,
  compilerOptions,
);
```

ここで重要なのは、この関数自身は

- 型解析
- コード補完
- JavaScript 出力

などを実装していないことである。

役割は、Language Service を利用できる状態まで 必要なオブジェクトを組み立てることにある。

---

## Builder としての役割

この関数を設計パターンの観点から見ると、Builder あるいは Composition Root に近い。

内部で行われる処理を簡略化すると、

```text
System
    │
    ▼

CompilerHost
    │
    ▼

LanguageServiceHost
    │
    ▼

LanguageService
    │
    ▼

VirtualTypeScriptEnvironment
```

となる。

それぞれの部品は、別々の責務を持っている。

Environment は、それらを接続し、利用しやすい形へまとめることだけを担当している。

---

# 5.3 LanguageService と Compiler の違い

ここで、Compiler と Language Service の違いを整理しておこう。

両者は同じ TypeScript Compiler API の一部であるが、目的が大きく異なる。

| Compiler                       | Language Service       |
| ------------------------------ | ---------------------- |
| コンパイルを行う               | エディタ支援を行う     |
| JavaScript を出力する          | コード補完を行う       |
| 型チェックを行う               | 定義ジャンプを行う     |
| 一度実行して終了することが多い | 編集中に何度も呼ばれる |

Compiler は、「コンパイルする」 ことが仕事である。

一方、Language Service は、「編集を支援する」 ことが仕事である。

Visual Studio Code や TypeScript Playground が利用しているのは、主に Language Service である。

---

## なぜ Language Service が必要なのか

例えば、ユーザーが

```ts
const value = Math.
```

まで入力したとする。

この時、エディタは

```
abs()

floor()

ceil()

max()

...
```

などの候補を表示する。

これは、Compiler の仕事ではない。

Language Service が、現在のカーソル位置や型情報を解析し、補完候補を返している。

同様に、

```
F12
```

による定義ジャンプや、リファクタリングも、Language Service が担当する。

そのため、ブラウザ上で TypeScript Playground のような環境を構築する場合、最も重要になるオブジェクトは Compiler ではなく、Language Service なのである。

---

# 5.4 LanguageServiceHost

Language Service を生成するためには、LanguageServiceHost というインターフェースが必要になる。

名前は、CompilerHost によく似ている。

しかし、両者の責務は異なる。

概念図で表すと、

```text
Compiler
      ▲
      │
CompilerHost


Language Service
      ▲
      │
LanguageServiceHost
```

となる。

つまり、どちらも Host ではあるが、接続先が異なる。

CompilerHost は、Compiler が必要とする情報を提供する。

一方、LanguageServiceHost は、Language Service が必要とする情報を提供する。

ここで重要なのは、Language Service は、Compiler よりも多くの情報を必要とすることである。

例えば、

- ファイルの現在の内容
- ファイルの更新回数
- プロジェクト内のファイル一覧
- CompilerOptions

など、エディタがリアルタイムに動作するための情報を管理する必要がある。

そのため、LanguageServiceHost は、CompilerHost よりも 「編集環境」 に近い役割を持っている。

---

## この章で理解できるようになったこと

- VirtualTypeScriptEnvironment は Language Service を利用するための実行環境である。
- `createVirtualTypeScriptEnvironment()` は部品を組み立てる Builder として機能する。
- Compiler と Language Service は役割が異なる。
- LanguageServiceHost は Language Service のための情報提供インターフェースである。
- Environment は TypeScript をブラウザ上で動作させるための土台となる。

---

# 第5章 VirtualTypeScriptEnvironment（前編 2/3）

前節では、VirtualTypeScriptEnvironment が Language Service を利用するための実行環境であること、そして、その中心には LanguageService が存在することを説明した。

しかし、Language Service は単独では動作できない。

Language Service が正常に動作するためには、常に現在のプロジェクト状態を取得できる仕組みが必要になる。

その役割を担うのが **LanguageServiceHost** である。

本節では、LanguageServiceHost がどのような情報を提供し、なぜ VirtualTypeScriptEnvironment がそれを管理しているのかを理解する。

---

# 5.5 LanguageServiceHost の責務

LanguageServiceHost は、Language Service が必要とする情報を取得するためのインターフェースである。

CompilerHost と同様に、Language Service は

直接 Virtual File System を操作しない。

代わりに、LanguageServiceHost を通して情報を取得する。

概念図で表すと、

```text
Language Service
        │
        ▼

LanguageServiceHost
        │
        ▼

VirtualTypeScriptEnvironment
        │
        ▼

Virtual File System
```

つまり、LanguageServiceHost は Language Service と Virtual File System の仲介役である。

---

## なぜ Host が必要なのか

Language Service は、できるだけ環境へ依存しないよう設計されている。

例えば、同じ Language Service でも、次のような環境で利用できる。

- Visual Studio Code
- TypeScript Playground
- Monaco Editor
- Node.js
- 独自エディタ

それぞれ、ファイル取得方法は異なる。

Language Service が 直接ブラウザ API や Node.js API を呼び出してしまうと、他の環境では動作できなくなる。

そこで、必要な情報だけを取得するための窓口として、LanguageServiceHost が存在する。

---

# 5.6 LanguageServiceHost が提供する情報

LanguageServiceHost が提供する情報は、CompilerHost より多い。

代表的なものとして、次のような情報がある。

| 情報                 | 用途                       |
| -------------------- | -------------------------- |
| ファイル一覧         | プロジェクト全体を把握する |
| ファイル内容         | 型解析を行う               |
| ファイルバージョン   | 更新されたか判定する       |
| CompilerOptions      | コンパイル条件を取得する   |
| カレントディレクトリ | モジュール解決に利用する   |

これらの情報を利用して、Language Service は 現在のプロジェクト状態を構築している。

---

## CompilerHost との違い

CompilerHost も ファイル内容を取得できた。

では、LanguageServiceHost と何が違うのだろうか。

最大の違いは、**更新** である。

Compiler は、通常

```
一度コンパイルする
```

という用途で利用される。

一方、Language Service は

```
編集する
↓
補完
↓
編集する
↓
型チェック
↓
編集する
↓
定義ジャンプ
```

という処理を、何百回、何千回と繰り返す。 そのため、「現在の状態」 を管理する仕組みが必要になる。

---

# 5.7 ファイルバージョン

Language Service を理解する上で、最も重要な概念の一つが **Version** である。

各ファイルには、変更回数を表す番号が存在する。

例えば、

```text
index.ts

version = 1
```

という状態から、ユーザーが

```ts
const a = 10;
```

を追加すると、

```text
version = 2
```

となる。

さらに、別の編集を行うと、

```text
version = 3
```

となる。

Language Service は、この Version を利用して、「どのファイルが変更されたか」 を判断する。

---

## なぜ Version が必要なのか

もし Version が存在しなければ、Language Service は 毎回

```
全ファイル
```

を解析し直す必要がある。

例えば、1000ファイルあるプロジェクトで、1文字だけ入力した場合でも、1000ファイル全てを再解析することになる。

これは非常に非効率である。

そこで、Version を利用することで、

```
変更されたファイルだけ
```

を再解析できる。

---

### コラム：TypeScript Playground が軽快に動く理由

Playground では、キーを押すたびに、補完や型エラーが更新される。

しかし、そのたびに プロジェクト全体を解析しているわけではない。

Language Service は、Version を見て、必要最小限だけを更新する。

この仕組みが、リアルタイム編集を支えている。

---

# 5.8 VirtualTypeScriptEnvironment の役割

ここで、Environment の役割をもう一度整理してみよう。

Environment 自身は、型解析を行わない。

コード補完もしない。

JavaScript 出力もしない。

これらは全て Language Service が担当する。

では、Environment は何をしているのだろうか。

答えは、**状態管理** である。

例えば、

```
ファイルを追加する
↓
Map を更新する
↓
Version を更新する
↓
Language Service が変更を検知する
```

という流れを成立させる。

つまり、Environment は Language Service の 「裏方」 として動作している。

---

# 5.9 updateFile() が存在する理由

Environment の代表的なメソッドとして、`updateFile()` がある。

名前だけを見ると、単純に

```
Map の値を書き換える
```

だけに思える。

しかし、実際にはそれ以上の役割を持っている。

概念的には、次のような処理が行われる。

```text
updateFile()

        │

        ├─ ファイル内容を更新する

        ├─ Version を増やす

        ├─ 必要な内部状態を更新する

        └─ Language Service が変更を認識できる状態にする
```

つまり、重要なのは Map を更新することではない。

Language Service が 「このファイルは更新された」 と認識できる状態を作ることである。

---

## 単純な Map 更新では駄目なのか

例えば、利用者が

```ts
fsMap.set("/index.ts", newSource);
```

だけを書いたとする。

Map の内容は更新される。

しかし、Language Service は それだけでは 変更を検知できない。

Version が変化していないためである。

その結果、古い解析結果を利用し続ける可能性がある。

そこで、利用者が Version 管理を意識しなくても済むよう、Environment が updateFile() という API を提供している。

これは、責務を利用者から隠蔽する、非常に重要な設計と言える。

---

## この節で理解できるようになったこと

- LanguageServiceHost は Language Service の情報提供窓口である。
- Language Service は Host を通して現在の状態を取得する。
- Version は変更検知のために利用される。
- VirtualTypeScriptEnvironment は状態管理を担当する。
- `updateFile()` は Map 更新だけでなく、Version 管理も行う。

---

# 第5章 VirtualTypeScriptEnvironment（前編 3/3）

前節では、VirtualTypeScriptEnvironment が Language Service の状態管理を担当していること、そして、`updateFile()` が単なる Map 更新ではなく、Language Service が変更を認識するための重要な API であることを説明した。

本節では、VirtualTypeScriptEnvironment が公開する代表的な操作を整理し、ライブラリ全体の設計思想をまとめる。

---

# 5.10 createFile()

仮想ファイルシステムでは、既存ファイルを書き換えるだけでなく、新しいファイルを追加したい場面がある。

例えば、Playground で

```
index.ts
```

だけだったプロジェクトへ、

```
math.ts
```

を追加するようなケースである。

概念的には、次のような流れになる。

```text
createFile()

        │

        ├─ 新しいファイルを Virtual File System へ追加

        ├─ Version を初期化

        ├─ プロジェクト状態を更新

        └─ Language Service が新しいファイルを認識する
```

ここでも重要なのは、Map に追加することではない。

Language Service が 「新しいファイルが増えた」 と認識できる状態を作ることである。

---

## なぜ `fsMap.set()` では足りないのか

Virtual File System は Map で管理されている。 そのため、一見すると、

```ts
fsMap.set("/math.ts", source);
```

だけで十分に思える。

しかし、Language Service は Map を直接監視しているわけではない。 内部では、LanguageServiceHost を通して、プロジェクト状態を取得している。

つまり、Map の更新だけでは、Language Service は ファイル追加を認識できない可能性がある。

Environment が `createFile()` という API を提供する理由は、この差異を吸収するためである。

---

# 5.11 deleteFile()

逆に、不要になったファイルを削除する場合は、`deleteFile()` が利用される。

概念的には、次のような処理となる。

```text
deleteFile()

        │

        ├─ Virtual File System から削除

        ├─ 管理情報を削除

        ├─ Project 状態を更新

        └─ Language Service へ反映
```

これも、Map の

```ts
delete()
```

だけでは十分ではない。

Language Service が 最新状態を認識できるよう、Environment が 必要な更新をまとめて行う。

---

# 5.12 emit()

TypeScript を利用する目的の一つが、JavaScript の生成である。 この処理を Compiler API では **Emit** と呼ぶ。

VirtualTypeScriptEnvironment でも、emit に相当する API が提供される。 概念的には、

```text
Source

        │

        ▼

Compiler

        │

        ▼

JavaScript
```

という流れになる。

ここで重要なのは、emit 自体は Environment が実装しているわけではない という点である。

実際に JavaScript を生成するのは、TypeScript Compiler である。 Environment は、Compiler を利用しやすい形で呼び出しているだけである。

---

## Environment は「代理人」

ここまで見てきた

```
updateFile()

createFile()

deleteFile()

emit()
```

は、どれも TypeScript Compiler の処理を 直接実装しているわけではない。

Environment は、利用者の代わりに 必要な情報を準備し、適切な順番で Compiler API や Language Service を呼び出している。

このような役割は、ソフトウェア設計では Facade（ファサード） と呼ばれることがある。

複雑な内部構造を隠し、利用者へ 分かりやすい API を提供する設計である。

VirtualTypeScriptEnvironment は、Builder として環境を構築し、Facade として環境を操作する。 二つの役割を兼ね備えていると言える。

---

# 5.13 VirtualTypeScriptEnvironment の設計思想

ここまで、個々の API を見てきた。

最後に、ライブラリ全体として、どのような思想で設計されているかを整理してみよう。

## 責務を分離する

まず特徴的なのは、責務が非常に細かく分割されていることである。

```text
Map

        │

        ▼

System

        │

        ▼

CompilerHost

        │

        ▼

LanguageServiceHost

        │

        ▼

LanguageService

        │

        ▼

VirtualTypeScriptEnvironment
```

一つのオブジェクトが 全てを担当するのではなく、各レイヤーが 一つの責務だけを持っている。

これは、Single Responsibility Principle（単一責任の原則） に沿った設計と言える。

---

## TypeScript 本体を再実装しない

もう一つ重要なのは、`@typescript/vfs` 自身は、型解析を実装していないことである。

コード補完も、型チェックも、JavaScript 出力も、全て TypeScript Compiler API が担当する。

VFS は、その API を ブラウザでも利用できるように 環境を整えているだけである。

つまり、ライブラリの価値は 「新しい Compiler を作ること」 ではなく、**既存の Compiler を利用しやすくすること** にある。

---

## 利用者へ複雑さを見せない

もし、Environment が存在しなければ、利用者は

```
CompilerHost

LanguageServiceHost

Version

Project

CompilerOptions

...
```

などを 全て理解しなければならない。

しかし、Environment があることで、利用者は

```ts
env.updateFile(...);

env.emit();
```

のような、シンプルな API だけを利用できる。

これは、ライブラリ設計として非常に大きな利点である。

---

# 5.14 この章のまとめ

本章では、VirtualTypeScriptEnvironment の役割と、ライブラリ全体における位置付けを説明した。 VirtualTypeScriptEnvironment は、Language Service を中心とした TypeScript の実行環境を表現するオブジェクトである。

内部では、

- Virtual File System
- System
- CompilerHost
- LanguageServiceHost
- Language Service

などを管理し、利用者へ統一された API を提供している。

また、

- `updateFile()`
- `createFile()`
- `deleteFile()`
- `emit()`

などの API は、単なるデータ操作ではなく、Language Service と整合性を保つための重要な役割を担っている。

---

## この章で理解できるようになったこと

- `createFile()` は Virtual File System と Language Service の状態を同期させる。
- `deleteFile()` は削除後のプロジェクト状態を管理する。
- `emit()` は TypeScript Compiler を利用して JavaScript を生成する。
- VirtualTypeScriptEnvironment は Builder と Facade の両方の性質を持つ。
- `@typescript/vfs` は TypeScript Compiler API を再実装するのではなく、利用しやすくするためのライブラリである。
- 第3章から第5章までで、Virtual File System から Language Service までの全体像を理解できる。
