# 第9章 `createVirtualTypeScriptEnvironment()` 完全読解（前編）

第8章までは、

`createVirtualTypeScriptEnvironment()` が何をする関数なのか、

そして、その設計思想について説明してきた。

本章からは、いよいよ実際のソースコードを読み進める。

ここからの目的は、

> **「ライブラリを使えるようになること」ではなく、「ライブラリを書いた作者の考え方を理解すること」**

である。

そのため、ソースコードを読む際には、

「この行は何をしているか」

だけでなく、

「なぜこのような設計になっているのか」

にも注目していく。

---

# 9.1 ソースコード全体を眺める

まずは関数全体を読む前に、

処理の流れだけを把握しておこう。

概略は次のようになっている。

```text
createVirtualTypeScriptEnvironment()

│

├─ CompilerHost を準備

├─ LanguageServiceHost を構築

├─ LanguageService を生成

└─ VirtualTypeScriptEnvironment を返す
```

ここで重要なのは、

**この関数自身は「何も解析していない」**

ということである。

例えば、

AST を生成する処理も、

型解析も、

コード補完も、

この関数には存在しない。

これらはすべて

TypeScript Compiler API

へ処理を委譲している。

つまり、

この関数は

**Builder**

あるいは

**Composition Root**

（オブジェクトを組み立てる場所）

として設計されている。

---

# 9.2 最初に注目するべきこと

ソースコードを読むと、

最初に目に入るのは

「Language Service を作るコード」

ではない。

その前に、

必要なオブジェクトを順番に準備している。

これは、

Language Service が

非常に多くの依存オブジェクトを必要とするためである。

例えば、

```
CompilerHost

CompilerOptions

RootFiles

System

TypeScript
```

などが揃っていなければ、

Language Service は生成できない。

そのため、

この関数では

「生成」

よりも

「準備」

のコードの方が多くなる。

---

# 9.3 `CompilerHost` を利用する理由

ここで、

第5章の内容を思い出そう。

```
Compiler
      ▲
      │
CompilerHost
      ▲
      │
System
```

という構造になっていた。

Language Service も、

Compiler と同じように

CompilerHost を利用する。

つまり、

第5章までで作ってきた仕組みは、

ここで初めて利用される。

これを見ると、

第3章から第5章までが

「準備編」

だったことが分かる。

---

## コラム：小さなライブラリほど「組み立て」が重要

`@typescript/vfs` のソースコードを読むと、

アルゴリズムらしいアルゴリズムはほとんど出てこない。

代わりに、

TypeScript Compiler API の部品を

どう組み合わせるか

というコードが続く。

これは、

近年のライブラリ設計ではよく見られるスタイルである。

ライブラリ自身が多くの機能を実装するのではなく、

既存ライブラリを適切に接続することで価値を提供している。

---

# 9.4 `LanguageServiceHost` の生成

次に、

Language Service のための Host が生成される。

ここで、

初心者が混乱しやすい点がある。

```
CompilerHost

LanguageServiceHost
```

は、

似た名前だが、

同じものではない。

両者の関係は、

次のようになる。

```text
Compiler
        ▲
        │
CompilerHost


LanguageService
        ▲
        │
LanguageServiceHost
```

つまり、

Host という名前は共通しているが、

対象となる API が異なる。

---

さらに、

LanguageServiceHost は

CompilerHost よりも多くの責務を持つ。

例えば、

- ファイルのバージョン
- スクリプト内容
- 編集後の状態

など、

エディタ特有の情報を管理する必要がある。

これが、

Language Service が

リアルタイム編集へ対応できる理由である。

---

# 9.5 `ts.createLanguageService()`

いよいよ、

TypeScript 本体の API が呼び出される。

ここで重要なのは、

Language Service は

`new LanguageService()`

のようなクラス生成ではない

ということである。

実際には、

Factory Function

である

```
ts.createLanguageService(...)
```

が利用される。

この設計は、

CompilerHost や System と同じ思想である。

TypeScript Compiler API では、

状態を持つオブジェクトを生成する場合でも、

公開 API は

Factory Function

として提供されることが多い。

これは、

初期化手順を利用者へ隠蔽し、

安全なオブジェクトだけを返すためである。

---

### コラム：Factory Function が多い理由

TypeScript Compiler API を見ていると、

```
createProgram()

createCompilerHost()

createLanguageService()

createSourceFile()
```

など、

`create` から始まる関数が非常に多い。

これは偶然ではない。

TypeScript 本体では、

利用者に

```
new ○○
```

を直接書かせるより、

Factory Function を経由させることで、

初期化方法を自由に変更できるよう設計されている。

そのため、

API の後方互換性も保ちやすい。

---

# 9.6 最後に返されるもの

Language Service が生成されたあと、

関数は

そのまま

Language Service

を返すわけではない。

返されるのは

```
VirtualTypeScriptEnvironment
```

である。

ここが、

このライブラリ最大の特徴と言える。

利用者が欲しいのは、

Language Service

だけではない。

例えば、

ブラウザ上では、

```text
ファイルを書き換える

↓

型エラーを更新する

↓

補完候補を更新する
```

という処理が必要になる。

これらを一つずつ呼び出すのは面倒である。

そこで作者は、

Language Service を中心に、

周辺機能もまとめた

Environment

というオブジェクトを設計した。

これによって、

利用者は

「TypeScript 開発環境」

そのものを扱えるようになる。

---

# 9.7 本章で見えてきた設計思想

ここまで読むと、

`createVirtualTypeScriptEnvironment()`

という関数名が、

非常によく考えられていることが分かる。

この関数は、

Language Service を返すだけではない。

CompilerHost や

LanguageServiceHost、

Virtual File System など、

複数の部品をまとめ、

一つの「開発環境」として返している。

つまり、

Environment

という命名は、

実装内容を正確に表しているのである。

---

## この章で理解できるようになったこと

- `createVirtualTypeScriptEnvironment()` は Builder（Composition Root）として機能する。
- 関数の役割は TypeScript Compiler API の部品を組み立てることである。
- `LanguageServiceHost` は CompilerHost よりもエディタ寄りの責務を持つ。
- `ts.createLanguageService()` は Factory Function として設計されている。
- 戻り値は Language Service 単体ではなく、`VirtualTypeScriptEnvironment` 全体である。
