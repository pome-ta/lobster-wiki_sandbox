# 第5章 `CompilerHost` の役割

第4章では、`ts.System` が **Compiler に対するファイルシステムの抽象化**であることを説明した。

しかし、ここで一つ疑問が生まれる。

> **`System` があるなら、Compiler はそのままコンパイルできるのではないか？**

実際には、それだけでは十分ではない。

TypeScript Compiler は、ファイルの内容を取得したあと、その文字列を解析し、AST を生成し、さらに型解析を行う必要がある。

この一連の処理を支えるのが **`CompilerHost`** である。

本章では、

- `CompilerHost` がなぜ必要なのか
- `System` との責務の違い
- `@typescript/vfs` における `createVirtualCompilerHost()` の役割

について解説する。

---

# 5.1 `System` だけでは足りない理由

第4章では、Compiler は

```text
System
```

を通して

```
readFile()
```

などを呼び出すことを説明した。

ここで考えてみよう。

例えば、

```ts
const source = system.readFile("/index.ts");
```

を実行すると、返ってくるのは

```ts
"const message = 'Hello';"
```

という **ただの文字列** である。

しかし、Compiler が解析する対象は文字列ではない。

Compiler が本当に欲しいものは、

```
SourceFile
```

である。

つまり、

```
文字列
    │
    ▼
AST
    │
    ▼
SourceFile
```

という変換が必要になる。

この責務は `System` には存在しない。

---

## 5.2 `SourceFile` とは

TypeScript Compiler は、ソースコードを直接解析し続けるわけではない。

まず、ソースコードを構文解析し、AST（Abstract Syntax Tree：抽象構文木）を生成する。

その AST を保持するオブジェクトが

```
SourceFile
```

である。

例えば、

```ts
const answer = 42;
```

というコードは、内部では概念的に次のような木構造になる。

```text
SourceFile
└── VariableStatement
    └── VariableDeclaration
        ├── Identifier ("answer")
        └── NumericLiteral (42)
```

以降、型解析や補完などは、この `SourceFile` を基に行われる。

つまり、Compiler が本当に扱う対象は **文字列ではなく `SourceFile`** なのである。

---

## 5.3 `CompilerHost` の責務

ここで `CompilerHost` が登場する。

役割を一言で表すと、

> **Compiler が `SourceFile` を取得するための窓口**

である。

概念図にすると、

```text
Compiler
      │
      ▼
CompilerHost
      │
      ├── System から文字列を取得
      │
      └── SourceFile を生成
```

となる。

つまり、`CompilerHost` は System の上位レイヤーである。

---

## 5.4 `System` と `CompilerHost` の違い

ここは非常に重要である。

両者は似ているようで、責務がまったく異なる。

### `System`

責務は

> **ファイルシステムとの通信**

例えば、

- ファイルを読む
- ファイルを書く
- ディレクトリを列挙する

など、OS に近い処理だけを担当する。

---

一方、

### `CompilerHost`

責務は

> **Compiler が必要とする情報を提供すること**

例えば、

- `SourceFile` を生成する
- 標準ライブラリを取得する
- 出力ファイルを書き込む
- Compiler の各種問い合わせに応答する

など、Compiler に近い役割を担当する。

---

つまり、責務は次のように分かれている。

```text
CompilerHost
        ↑
Compiler の都合

---------------------

System
        ↑
OS の都合
```

この分離によって、ファイルシステムが変わっても、CompilerHost はほとんど変更せずに済む。

逆に、CompilerHost の実装を変更しても、System は影響を受けない。

これは責務分離（Separation of Concerns）の典型的な例である。

---

## 5.5 なぜ二段階に分かれているのか

初心者が最も疑問に思う点はここである。

```
System
```

だけでも、`SourceFile` を返せば良いように思える。

しかし、そうすると System は

```
TypeScript
```

という言語を知る必要が出てしまう。

例えば、JavaScript を解析するのか、TypeScript を解析するのか、JSX を解析するのか、という知識まで持つことになる。

これは System の責務ではない。 System は あくまで

```
ファイル
```

だけを扱う。

その上で、CompilerHost が

```
TypeScript の世界
```

へ橋渡しを行う。

つまり、
- System は **汎用的なファイルシステム**
- CompilerHost は **TypeScript 専用のレイヤー**

という役割分担になっている。



---

### コラム：CompilerHost は「橋」である

これまでの構造を整理すると、

```
Map
    │
    ▼
System
    │
    ▼
CompilerHost
    │
    ▼
Compiler
```

となる。

このうち、Map と System は「ファイルシステム側」の世界である。

一方、Compiler は「TypeScript の世界」である。

CompilerHost は、この二つの世界をつなぐ橋として設計されている。

この橋があることで、Compiler は ファイルシステムの実装を意識する必要がなくなり、System は TypeScript の構文を理解する必要もなくなる。

---

## 5.6 `createVirtualCompilerHost()`

`@typescript/vfs` が提供する

```
createVirtualCompilerHost()
```

は、

この CompilerHost を構築するための関数である。

入力は、概念的には

```
System
```

であり、出力は

```
CompilerHost
```

となる。

つまり、

```
Map
    │
    ▼
System
    │
    ▼
CompilerHost
```

という変換を担当している。

この関数自身が解析を行うわけではない。

あくまで、TypeScript Compiler が利用できる `CompilerHost` を組み立てることが責務である。

---

## この章のまとめ

本章では、`CompilerHost` の役割を説明した。 重要な点は次のとおりである。

- Compiler が扱うのは文字列ではなく `SourceFile` である。
- `System` はファイルシステムとの通信だけを担当する。
- `CompilerHost` は Compiler に必要な情報を提供する。
- `CompilerHost` は System と Compiler の橋渡しを行う。
- `createVirtualCompilerHost()` は Virtual File System 用の `CompilerHost` を構築する。
