# 第 2 章 アーキテクチャ

本章では、`@typescript/vfs` のソースコード全体を俯瞰し、

- どのようなディレクトリ構成になっているのか
- 各ソースファイルは何を担当しているのか
- 利用者が API を呼び出したとき、内部ではどのような処理が行われるのか

を解説する。

本章は個々の関数の詳細ではなく、ライブラリ全体の構造を理解することを目的とする。

---

## 2.1 ディレクトリ構成

`packages/typescript-vfs` は、大まかに次のような構成になっている。

```text
typescript-vfs/
├── src/
│   ├── index.ts
│   ├── createSystem.ts
│   ├── createVirtualCompilerHost.ts
│   ├── createVirtualTypeScriptEnvironment.ts
│   ├── createDefaultMapFromCDN.ts
│   ├── createDefaultMapFromNodeModules.ts
│   ├── ...
│
├── package.json
├── README.md
└── tsconfig.json
```

実際のファイル名は今後変更される可能性があるが、役割としては概ねこのように分割されている。

特徴として、**1 ファイル = 1 つの責務** という構成になっている。（これは、嘘で`index.ts` にまとめてある）

巨大なクラスや複雑な継承はなく、必要な関数だけをエクスポートする設計になっている。

---

## 2.2 モジュール構成

各モジュールの役割を整理すると、次のようになる。

| モジュール                           | 役割                                                     |
| ------------------------------------ | -------------------------------------------------------- |
| `index.ts`                           | 公開 API をまとめてエクスポートする                      |
| `createSystem`                       | 仮想ファイルシステムを `ts.System` として公開する        |
| `createVirtualCompilerHost`          | CompilerHost を構築する                                  |
| `createVirtualTypeScriptEnvironment` | Language Service を利用できる環境を構築する              |
| `createDefaultMapFromCDN`            | 標準ライブラリ (`lib.dom.d.ts` など) を CDN から取得する |
| `createDefaultMapFromNodeModules`    | Node.js 上で標準ライブラリを読み込む                     |

ライブラリ全体として見ると、

ほぼすべての API は

```text
Environment
        │
        ▼
CompilerHost
        │
        ▼
System
```

という階層構造になっている。

つまり、高レベル API は低レベル API を組み合わせて作られている。

---

## 2.3 レイヤー構造

役割ごとに分けると、`@typescript/vfs` は次のようなレイヤー構造を持つ。

```text
┌────────────────────────────────────┐
│ createVirtualTypeScriptEnvironment │
└──────────────┬─────────────────────┘
               │
               ▼
┌────────────────────────────┐
│ createVirtualCompilerHost  │
└──────────────┬─────────────┘
               │
               ▼
┌────────────────────────────┐
│ createSystem               │
└──────────────┬─────────────┘
               │
               ▼
┌────────────────────────────┐
│ Map<string, string>        │
└────────────────────────────┘
```

一番下にあるのは、ただの `Map` である。

そこへ`createSystem()` が

```text
Map
↓

ts.System
```

へ変換する。

さらに、`createVirtualCompilerHost()` が

```text
ts.System
↓

CompilerHost
```

を作成する。

最後に、`createVirtualTypeScriptEnvironment()` が

```text
CompilerHost
↓

Language Service
```

まで組み立てる。

つまり、利用者が普段使う API は、実は何段階もの初期化処理を内部で行っている。

---

## 2.4 データフロー

例えば、利用者が

```ts
env.updateFile("/index.ts", source);
```

を呼び出したとする。

内部では概ね次のような流れになる。

```text
利用者
    │
    ▼
updateFile()
    │
    ▼
Map の内容を書き換える
    │
    ▼
ScriptVersion を更新する
    │
    ▼
Language Service が変更を検知
    │
    ▼
必要な SourceFile を再解析
```

重要なのは、**毎回すべてのファイルを解析し直すわけではない** ということである。

TypeScript Language Service はインクリメンタル解析を行うため、変更されたファイルだけを効率よく再解析する。

そのため、エディタ上で文字を 1 文字入力するたびに、プロジェクト全体を解析し直しているわけではない。

---

## 2.5 依存関係

モジュール同士の依存関係は比較的単純である。

```text
createVirtualTypeScriptEnvironment
                │
                ▼
createVirtualCompilerHost
                │
                ▼
createSystem
                │
                ▼
Map
```

逆方向の依存は存在しない。

つまり、下位レイヤーは上位レイヤーの存在を知らない。

例えば、`createSystem()` は Language Service が存在することを知らない。

単に

> 「Map を ts.System として扱えるようにする」

だけである。

このような設計により、必要な機能だけを利用できる。

例えば、CompilerHost が不要であれば`createSystem()` だけを利用することも可能である。

---

## 2.6 なぜ関数中心の設計なのか

ソースコードを読むと、大きなクラスはほとんど登場しない。

代わりに、

```ts
createSystem();

createVirtualCompilerHost();

createVirtualTypeScriptEnvironment();
```

のような**Factory Function** が中心となっている。

これは TypeScript Compiler API 自体が、インターフェースとオブジェクトリテラルを多用する設計だからである。

例えば、`ts.System` も

```ts
interface System
```

として定義されており、クラスではない。

そのため、VFS でも

```ts
const system = {
    readFile() {},
    writeFile() {},
    ...
}
```

というオブジェクトを生成して返す設計になっている。

これは TypeScript 本体の設計思想に合わせた実装と言える。

---

## この章のまとめ

本章では、`@typescript/vfs` 全体の構造を確認した。

重要な点は次のとおりである。

- ライブラリは責務ごとにモジュールが分割されている。
- 高レベル API は低レベル API を組み合わせて構築される。
- 最終的には `Map<string, string>` を TypeScript Compiler が扱える形へ変換している。
- クラスではなく、Factory Function とインターフェースを中心とした設計になっている。
