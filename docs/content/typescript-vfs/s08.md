# 第8章 `createVirtualTypeScriptEnvironment()` の実装を読む（前編）

第7章では、`createVirtualTypeScriptEnvironment()` の関数シグネチャから設計思想を読み解いた。

本章からは、いよいよ関数本体へ入っていく。

ここで一つ意識しておきたいことがある。

**この関数は、新しいアルゴリズムを実装しているわけではない。**

むしろ逆である。

作者がやっていることは、

```
TypeScript Compiler API が提供している部品
```

を、決められた順番で組み立てているだけである。

そのため、ソースコードを読む際には、

> 「何か新しい処理を実装しているか」

ではなく、> **「どの TypeScript API を、どの順番で呼び出しているか」** を見ることが重要になる。

---

# 8.1 関数本体は「組み立て処理」

一般的なライブラリでは、関数本体の中で

- 独自アルゴリズム
- 独自データ構造
- 複雑な計算

などが実装されていることが多い。

しかし、`@typescript/vfs` は違う。 役割は

```
Map
↓
System
↓
CompilerHost
↓
LanguageServiceHost
↓
LanguageService
```

という既存の部品を接続することである。

つまり、作者の仕事は 「作る」 ではなく 「繋ぐ」 なのである。

---

# 8.2 最初に行われること

関数本体では、まず Language Service を構築するために必要な部品が準備される。

ここで重要なのは、Language Service は 単独では生成できない という点である。

必要になるのは、少なくとも

- CompilerHost
- CompilerOptions
- Root Files
- TypeScript Compiler API

などである。

つまり、Environment は それらを一つにまとめる オーケストレーター として振る舞う。

---

### コラム：オーケストレーターとは

ソフトウェア設計では、複数の部品を組み合わせるだけのクラスや関数を **Orchestrator（オーケストレーター）** と呼ぶことがある。

自分では大きな処理を行わず、各部品へ処理を依頼する。 `createVirtualTypeScriptEnvironment()` も、まさにその役割を担っている。

---

# 8.3 `LanguageServiceHost` が登場する

ここで、新しい名前が現れる。

```
LanguageServiceHost
```

である。

名前は CompilerHost によく似ている。

実際、役割も近い。

違いは、対象となる相手である。

```
CompilerHost
        │
        ▼
Compiler
```

に対し、

```
LanguageServiceHost
        │
        ▼
LanguageService
```

となる。

つまり、CompilerHost が Compiler のための窓口だったように、LanguageServiceHost は Language Service のための窓口なのである。

---

# 8.4 なぜ Host が増えるのか

ここで疑問が生まれる。

> 「CompilerHost があるなら、それをそのまま使えばよいのではないか？」

実際には、Language Service は Compiler より多くの情報を必要とする。

例えば、エディタでは

```
ユーザーが文字を1文字入力
↓
補完候補を更新
↓
型エラーを更新
↓
ジャンプ先を更新
```

という処理が何度も繰り返される。

そのため、Language Service は 「現在のファイル状態」 を常に知っている必要がある。

CompilerHost は、一度コンパイルするための情報提供が中心である。

一方、LanguageServiceHost は、**変化し続ける編集状態** を管理することが目的となる。

---

# 8.5 バージョン管理という考え方

Language Service では、各ファイルに **Version** という概念が存在する。 例えば、

```
version = 1
```

だったファイルへ 文字を入力すると、

```
version = 2
```

となる。

さらに編集すると、

```
version = 3
```

となる。

Language Service は この Version を見て、「このファイルは変更された」 と判断する。

これにより、変更されていないファイルまで 毎回解析し直す必要がなくなる。

---

### コラム：Playground が高速な理由

TypeScript Playground では、キーを1文字入力するたびに、補完やエラー表示が更新される。

しかし、毎回プロジェクト全体を解析しているわけではない。

Language Service は Version を利用し、変更されたファイルだけを再解析している。

この仕組みによって、リアルタイム編集が実現されている。

---

# 8.6 `@typescript/vfs` の役割

ここで、再び `@typescript/vfs` の立場を整理してみよう。

Language Service 自体は、TypeScript 本体が提供している。

Version 管理の仕組みも、TypeScript 本体で実装されている。

では、VFS は何をしているのか。 答えは、それらを利用できるように 必要な情報を提供しているだけである。

つまり、

```
VFS
        │
        ▼
LanguageServiceHost
        │
        ▼
LanguageService
```

という橋渡しを担当している。

ここでも、ライブラリ自身が 型解析を実装しているわけではない。

---

# 8.7 この章で見えてきたこと

ここまで読んで分かるのは、`createVirtualTypeScriptEnvironment()` の中心は Language Service そのものではなく、**Language Service を成立させるための土台** であるということである。

第3章から積み上げてきた

```
Map
↓
System
↓
CompilerHost
```

は、すべて この地点へ到達するための準備だったのである。

---

## この章で理解できるようになったこと

- `createVirtualTypeScriptEnvironment()` はオーケストレーターとして振る舞う。
- Language Service は単独では生成できず、多くの部品を必要とする。
- `LanguageServiceHost` は Language Service のための窓口である。
- ファイルの Version 管理はリアルタイム編集を支える重要な仕組みである。
- `@typescript/vfs` は Language Service を実装するのではなく、それを利用できる環境を整えている。
