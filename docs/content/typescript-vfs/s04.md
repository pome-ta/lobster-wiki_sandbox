# 第4章 `createSystem()` の実装

第3章では、

> `Map<string, string>` を `ts.System` へ変換する

という概要を説明した。

本章では、実際に `createSystem()` がどのような設計になっているのかをソースコードに沿って見ていく。

ここからは、「TypeScript Compiler は何を要求しているのか」という視点で読むと理解しやすい。

---

# 4.1 `ts.System` はインターフェース

まず重要なのは、`ts.System` はクラスではない。

TypeScript 本体では概ね次のようなインターフェースになっている。

```ts
interface System {

    readFile(...)

    writeFile(...)

    fileExists(...)

    directoryExists(...)

    readDirectory(...)

    getCurrentDirectory(...)

    useCaseSensitiveFileNames

    newLine

    ...
}
```

つまり、

Compiler は

```ts
system.readFile(...)
```

のようなメソッドを持つオブジェクトであれば利用できる。

継承も不要。`new System()` のようなクラス生成も存在しない。

---

# 4.2 `createSystem()` はオブジェクトを返す

そのため`createSystem()` は、概念的には次のような実装になっている。

```ts
return {

    readFile() {
        ...
    },

    writeFile() {
        ...
    },

    fileExists() {
        ...
    }

}
```

つまり、クラスではなく **オブジェクトリテラル** を返している。

これは TypeScript Compiler API 全体でもよく見られる設計である。

---

# 4.3 `readFile()`

最も重要なのが

```ts
readFile(path)
```

である。

Compiler はSourceFile を生成する前に、必ずここを通る。

概念的には

```ts
readFile(path) {

    return fsMap.get(path);

}
```

となる。

つまり、Compiler が

```ts
readFile("/index.ts")
```

を呼ぶと、内部では

```ts
fsMap.get("/index.ts")
```

が実行されるだけである。

---

存在しない場合は、通常

```ts
undefined
```

が返る。

Compiler はこれを見て

> ファイルが存在しない

と判断する。

---

# 4.4 `writeFile()`

書き込みはさらに単純である。

概念的には

```ts
writeFile(path, text) {

    fsMap.set(path, text);

}
```

となる。

つまり、JavaScript を emit した場合も、本当のファイルへ保存されるわけではない。

例えば

```text
/index.js
```

が生成されたとしても、実際には

```text
Map
```

へ追加されるだけである。

そのため、ブラウザでもemit が可能になる。

---

# 4.5 `fileExists()`

Compiler は頻繁に

```ts
fileExists()
```

を呼ぶ。

これは

```ts
readFile()
```

より先に呼ばれることも多い。

実装は概ね

```ts
fileExists(path) {

    return fsMap.has(path);

}
```

となる。

つまり、Map にキーが存在するかどうかだけを見ている。

---

# 4.6 `directoryExists()`

ここは少し特殊である。

Map には

```text
ディレクトリ
```

という概念が存在しない。

あるのは

```text
"/src/index.ts"
```

という文字列だけである。

そのため、

```ts
directoryExists("/src")
```

が呼ばれた場合、Map のキーを調べて

```text
"/src/"
```

で始まるファイルが存在するかを確認する。

つまり、ディレクトリを保持しているのではなく、 **ファイルパスからディレクトリを推測している。**

---

# 4.7 `readDirectory()`

これもMap を走査するだけである。

例えば、

```text
/src/index.ts

/src/util.ts

/test/main.ts
```

が存在するとする。

すると、

```ts
readDirectory("/src")
```

は

```text
/src/index.ts

/src/util.ts
```

だけを返す。

つまり、Map 全体を検索し、条件に一致するものを抽出している。

---

# 4.8 `getCurrentDirectory()`

Compiler は相対パスを解決するため、現在のディレクトリを知る必要がある。

例えば

```text
.
```

や

```text
..
```

を解決するためである。

そのためSystem は

```ts
getCurrentDirectory()
```

を提供する。

Virtual File System でも現在位置という概念だけは持っている。

---

# 4.9 `useCaseSensitiveFileNames`

Windows は

```text
Index.ts

index.ts
```

を同じファイルとして扱う。

一方、Linux は別ファイルである。

Compiler はこの違いを知る必要がある。

そのためSystem は

```ts
useCaseSensitiveFileNames
```

を持っている。

Virtual File System でもこの情報を返すことで、Compiler は各 OS と同じ挙動を再現できる。

---

# 4.10 `newLine`

Compiler がemit を行う際、改行コードを知る必要がある。

そのため、

```ts
newLine
```

も System が提供する。

例えば

```text
"\n"
```

や

```text
"\r\n"
```

などである。

---

# 4.11 `createSystem()` は状態を持つ

ここまで見ると、`createSystem()` は単なる変換関数に思える。

しかし、実際には内部に

```text
Map
```

を保持している。

つまり、返された System は同じ Map を共有し続ける。

例えば

```ts
system.writeFile(...)
```

を実行すると、その後

```ts
system.readFile(...)
```

で同じ内容が取得できる。

これは同じ Map を参照しているからである。

つまり、System 自体がVirtual File System の窓口になっている。

---

# 4.12 この設計の利点

この設計により、Compiler はVirtual File System の存在を全く知らない。

Compiler が見ている世界は

```text
Compiler
      │
      ▼
System
```

だけである。

その下に

```text
Map

IndexedDB

localStorage

HTTP

ZIP

メモリ
```

など、

何が存在していても構わない。

System が同じインターフェースを実装していれば、Compiler はそのまま利用できる。

これがTypeScript Compiler API の柔軟さを支えている。

---

# この章のまとめ

本章では

`createSystem()` が

`Map` を

`ts.System`

へ変換する実装を見た。

重要な点は次のとおりである。

- `ts.System` はクラスではなくインターフェースである。
- `createSystem()` はオブジェクトリテラルを返す。
- 各メソッドは最終的に `Map` を操作している。
- Compiler は `System` のみを見ており、保存先が何であるかは意識しない。

---

# 第4章 `ts.System` と `createSystem()`

第3章では、Virtual File System の実体が `Map<string, string>` であることを説明した。

しかし、その章の最後で触れたように、

> `Map` をそのまま TypeScript Compiler に渡しても動作しない。

これは非常に重要なポイントである。

本章では、その理由を説明し、

`Map` を TypeScript Compiler が利用できる形へ変換する **`ts.System`** と **`createSystem()`** の役割について解説する。

---

## 4.1 なぜ `Map` を直接渡せないのか

第3章では、Virtual File System を次のように表現した。

```ts
const fsMap = new Map([
    ["/index.ts", "const message = 'Hello';"],
]);
```

JavaScript の視点では、

これは十分にファイルを表現できている。

例えば、

```ts
fsMap.get("/index.ts");
```

とすれば、

ファイル内容を取得できる。

しかし、

TypeScript Compiler は

```ts
fsMap.get(...)
```

という API を知らない。

Compiler が知っているのは、

```ts
system.readFile(...)

system.fileExists(...)

system.readDirectory(...)
```

などである。

つまり、

Compiler が期待しているインターフェースと、

`Map` が提供するインターフェースは一致していない。

この「形の違い」を吸収する必要がある。

---

## 4.2 `ts.System` とは

TypeScript Compiler は、

OS を直接操作しない。

例えば、

Node.js であれば

```ts
import fs from "node:fs";
```

を利用すればファイルを読める。

しかし、

TypeScript Compiler のソースコードは、

そのような Node.js 固有の API を直接呼び出さない。

代わりに、

常に

```text
ts.System
```

という抽象インターフェースを経由する。

概念的には、

次のような構造になっている。

```text
Compiler
      │
      ▼
  ts.System
      │
      ▼
実際のファイルシステム
```

つまり、

Compiler は

「どの OS 上で動いているか」

を知らない。

知っているのは

```
System
```

だけである。

これは、

TypeScript を

- Windows
- macOS
- Linux
- ブラウザ
- Deno

など、

様々な環境で動作させるための重要な設計である。

---

## 4.3 `ts.System` は「OS の代理人」

`ts.System` を一言で表現すると、

> **Compiler に対する OS の代理人**

である。

例えば、

Compiler が

```ts
readFile("/index.ts")
```

を呼び出したとする。

Node.js 版では、

内部で最終的に

```ts
fs.readFileSync(...)
```

が呼ばれる。

一方、

Virtual File System では、

最終的に

```ts
Map.get(...)
```

が呼ばれる。

つまり、

同じ

```ts
readFile()
```

という API でも、

実際の処理は環境によって異なる。

Compiler は

その違いを一切意識しない。

---

### コラム：これは「Adapter パターン」に近い

オブジェクト指向の設計では、

異なる API を共通インターフェースへ変換する手法を

**Adapter パターン**

と呼ぶ。

```
Map
    │
    ▼
createSystem()
    │
    ▼
ts.System
```

という構造は、

まさに Adapter の役割を果たしている。

`createSystem()` は、

Map を変更するのではなく、

Map を利用するための「通訳」を作っているのである。

---

## 4.4 `createSystem()` の役割

`createSystem()` の責務は非常に明確である。

> **Virtual File System を `ts.System` として公開すること**

である。

図にすると、

```text
Map<string, string>
        │
        ▼
createSystem()
        │
        ▼
ts.System
```

となる。

ここで重要なのは、

`createSystem()` は

新しいファイルシステムを作る関数ではない

ということである。

Virtual File System の実体は、

あくまで `Map` である。

`createSystem()` は、

その `Map` を利用するためのインターフェースを生成しているだけである。

---

## 4.5 `ts.System` はクラスではない

ここで、

TypeScript Compiler API の設計思想が現れる。

多くのライブラリであれば、

次のようなクラスを想像するかもしれない。

```ts
const system = new System();
```

しかし、

TypeScript Compiler API は

このような設計になっていない。

`ts.System` は

**インターフェース**

として定義されている。

つまり、

必要なのは

```
readFile()

writeFile()

fileExists()

...
```

というメソッドを持つオブジェクトである。

したがって、

`createSystem()` は

クラスを生成するのではなく、

概念的には

```ts
return {

    readFile() {},

    writeFile() {},

    fileExists() {},

}
```

のようなオブジェクトを返している。

これは TypeScript Compiler API 全体で採用されている設計思想であり、

継承よりも

**インターフェースを実装したオブジェクト**

を重視している。

---

## 4.6 `createSystem()` は状態を持つ

「オブジェクトを返すだけ」と聞くと、

単なるユーティリティ関数のように感じるかもしれない。

しかし、

返された `System` は、

Virtual File System を保持し続ける。

例えば、

```ts
system.writeFile("/index.ts", source);
```

を実行した後、

```ts
system.readFile("/index.ts");
```

を呼び出すと、

同じ内容が取得できる。

これは、

`System` が

同じ `Map` を内部で参照しているからである。

つまり、

`createSystem()` は

単なる変換関数ではなく、

Virtual File System への窓口を生成しているのである。

---

## 4.7 `System` は Compiler 全体の土台

ここまでを見ると、

`System` は

単なるファイル操作 API のようにも見える。

しかし、

TypeScript Compiler 全体で見ると、

`System` は

最も下位の抽象レイヤーである。

```
Compiler
      │
      ▼
CompilerHost
      │
      ▼
System
      │
      ▼
OS または Virtual File System
```

この構造を見ると、

Compiler は

OS を知らない。

CompilerHost も

OS を知らない。

実際に OS と通信するのは、

`System`

だけである。

つまり、

**`System` は TypeScript Compiler 全体の基盤となる抽象化**

なのである。

この設計のおかげで、

TypeScript は

実際の OS だけでなく、

ブラウザ上の Virtual File System に対しても、

全く同じ Compiler を利用できる。

---

## この章のまとめ

本章では、

`ts.System` と `createSystem()` の役割を説明した。

重要な点は次のとおりである。

- `Map` は Compiler が理解できる形式ではない。
- `ts.System` は Compiler に対するファイルシステムの共通インターフェースである。
- `createSystem()` は `Map` を `ts.System` として公開する Adapter の役割を持つ。
- `ts.System` はクラスではなくインターフェースであり、オブジェクトリテラルで実装できる。
- `System` は TypeScript Compiler 全体の最下層を支える重要な抽象レイヤーである。
