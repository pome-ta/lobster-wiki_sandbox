# 第 3 章 (1 稿) Virtual File System

本章では、`@typescript/vfs` の中核となる **Virtual File System** について解説する。

この章では、

- なぜ `Map` が採用されているのか
- `Map` はどのようにファイルシステムとして扱われるのか
- `createSystem()` は何を生成するのか

を理解することを目的とする。

---

# 3.1 なぜ `Map` を使うのか

Virtual File System の実体は、

```ts
Map<string, string>;
```

である。

一見すると非常に単純で、

```ts
const fsMap = new Map([
  ["/index.ts", "const x = 1;"],
  ["/util.ts", "export const y = 2;"],
]);
```

これだけである。

キーは

```text
ファイルパス
```

値は

```text
ファイル内容
```

となる。

つまり、

```text
/index.ts
    ↓
const x = 1;
```

という対応表を持っているだけである。

---

## なぜオブジェクトではないのか

例えば、

```ts
const files = {
  "/index.ts": "...",
  "/util.ts": "...",
};
```

でも実現できそうに見える。

しかし `Map` には次の利点がある。

### キーに任意の文字列を使える

```ts
map.set("/src/index.ts", source);
```

パスをそのままキーにできる。

---

### 順序が保証される

`Map` は追加順を保持する。

Compiler は順序に依存しないが、デバッグや列挙時には便利である。

---

### API が豊富

```ts
map.has(path);

map.get(path);

map.set(path, text);

map.delete(path);
```

など、ファイルシステムに必要な操作が揃っている。

---

### 更新コストが小さい

1 ファイルだけ更新したい場合、

```ts
map.set(path, newText);
```

だけで済む。

新しいオブジェクトを作り直す必要はない。

---

そのため、`@typescript/vfs` では最初から `Map` を Virtual File System の実体として採用している。

---

# 3.2 Virtual File System は「ファイルシステム」ではない

ここで重要なのは、`Map` 自体は **ファイルシステムではない** ということである。

例えば、Compiler は

```ts
readFile(path);
```

のような API を期待している。

しかし、`Map` には

```ts
readFile();
```

は存在しない。

できるのは

```ts
map.get(path);
```

だけである。

つまり、Compiler は

```text
Map
```

を直接扱えない。

そこで登場するのが

```ts
createSystem();
```

である。

---

# 3.3 `createSystem()` の役割

`createSystem()` は、

```text
Map
```

を

```text
ts.System
```

へ変換する関数である。

概念図で表すと、

```text
Map
      │
      ▼
createSystem()
      │
      ▼
ts.System
```

となる。

つまり、Virtual File System を TypeScript Compiler が理解できる形へ変換している。

---

# 3.4 `ts.System` とは

`ts.System` は、

TypeScript Compiler が「OS」と通信するためのインターフェースである。

例えば、Compiler は

```ts
system.readFile(path);
```

や

```ts
system.fileExists(path);
```

を呼び出す。

つまり Compiler は、実際の OS を知らない。

知っているのは

```text
System
```

だけである。

Node.js では、この `System` が内部で

```text
fs
```

を利用している。

一方、`@typescript/vfs` では内部で

```text
Map
```

を利用している。

Compiler から見ると、どちらも

```text
ts.System
```

なので違いは分からない。

---

# 3.5 `createSystem()` が実装するもの

`createSystem()` は`Map` を利用して`ts.System` の各メソッドを実装する。

例えば、

```ts
readFile(path);
```

は概念的には

```ts
return fsMap.get(path);
```

となる。

---

```ts
fileExists(path);
```

なら、

```ts
return fsMap.has(path);
```

で判定できる。

---

```ts
writeFile(path, text);
```

なら、

```ts
fsMap.set(path, text);
```

になる。

つまり、実際には Map を操作しているだけである。

Compiler は

```text
readFile()
```

を呼んでいるつもりだが、内部では

```text
Map.get()
```

が実行されている。

---

# 3.6 実際に仮想化されるもの

`createSystem()` が扱うのは、単なる

```text
ファイル内容
```

だけではない。

Compiler は次のような情報も必要とする。

- ファイルの存在確認
- ディレクトリの存在確認
- ディレクトリ一覧
- カレントディレクトリ
- 改行コード
- 大文字小文字を区別するか

など。

つまり、Virtual File System は

```text
Map
```

だけでは完成しない。

`createSystem()` はこれらの情報も含めて「仮想 OS」を作っている。

もちろん、本物の OS のようにプロセス管理やアクセス権限まで持っているわけではない。

Compiler が必要とする最低限の機能だけを提供している。

---

# 3.7 なぜ `System` を作るだけなのか

一見すると、

```ts
readFile(path) {
    return map.get(path);
}
```

程度なら、Compiler が直接 `Map` を扱えばよさそうに思える。

しかし、TypeScript Compiler は

- Node.js
- Deno
- ブラウザ
- 将来の新しい環境

など、様々な環境で動作することを想定している。

そのため、Compiler は

```text
System
```

だけに依存する設計になっている。

つまり、新しい実行環境を追加したい場合は、新しい `System` を実装するだけで済む。

Compiler 本体を変更する必要はない。

これは **依存性逆転（Dependency Inversion）** の考え方に近い設計になっている。

---

# この章のまとめ

本章では Virtual File System の基礎を説明した。

重要な点は次のとおりである。

- Virtual File System の実体は `Map<string, string>` である。
- `Map` 自体は Compiler が理解できない。
- `createSystem()` が `Map` を `ts.System` に変換する。
- Compiler は `System` のみを利用し、実際の保存先が OS なのか `Map` なのかを意識しない。

---

# 第 3 章(2 稿) Virtual File System

第 1 章では、`@typescript/vfs` が「TypeScript Compiler API をブラウザなどのファイルシステムを持たない環境で利用するためのライブラリ」であることを説明した。

第 2 章では、その全体構造を俯瞰し、

```
Map
    ↓
System
    ↓
CompilerHost
    ↓
Language Service
```

というレイヤー構造を確認した。

本章からは、その最下層である **Virtual File System** を詳しく見ていく。

---

## 3.1 Virtual File System とは何か

Virtual File System（以下 VFS）は、日本語では「仮想ファイルシステム」と訳される。

しかし、この名前だけを見ると、

> 「OS のファイルシステムを丸ごとエミュレートするもの」

という印象を受けるかもしれない。

`@typescript/vfs` が実装しているものは、それとは少し異なる。

実際には、

> **TypeScript Compiler が必要とする最小限のファイルシステム**

を実装している。

つまり、

```
Windows

macOS

Linux
```

のような OS を再現しているわけではない。

Compiler が

- ファイルを読む
- ファイルを書く
- ディレクトリを列挙する

ために必要な機能だけを提供している。

そのため、本書では以降、「Virtual File System」という言葉を

> **TypeScript Compiler のための仮想ファイルシステム**

という意味で使用する。

---

## 3.2 なぜ Virtual File System が必要なのか

この疑問に答えるには、まず TypeScript Compiler がどのように動作するのかを知る必要がある。

例えば、次のようなプロジェクトがあるとする。

```text
project/
├── src/
│   ├── index.ts
│   └── util.ts
├── tsconfig.json
└── package.json
```

開発者が

```bash
tsc
```

を実行すると、Compiler は概ね次のような処理を行う。

1. `tsconfig.json` を読む
2. コンパイル対象のファイルを探す
3. `index.ts` を読む
4. `util.ts` を読む
5. `lib.dom.d.ts` など標準ライブラリを読む
6. AST を生成する
7. 型解析する
8. JavaScript を出力する

ここで重要なのは、 **最初の段階で大量のファイルアクセスが発生している** ということである。

Compiler は

```
ファイルを読む
```

ことを前提として設計されている。

つまり、

```
Compiler
    ↓
File System
```

という依存関係を持っている。

---

ところが、ブラウザには

```
C:\

/home/

/Users/
```

のようなファイルシステムは存在しない。

JavaScript が扱えるのは、基本的に

- 文字列
- オブジェクト
- 配列
- Map

などのメモリ上のデータだけである。

このままでは、Compiler は

```
readFile()
```

を実行できない。

つまり、ブラウザでは TypeScript Compiler を動かせないことになる。

---

ここで登場するのが Virtual File System である。

Virtual File System は、Compiler が期待する

```
ファイル
```

という概念を、JavaScript のデータ構造だけで実現する。

その結果、Compiler は「ブラウザで動いている」ことを意識せず、通常のファイルシステムを扱うのと同じように動作できる。

---

## 3.3 Virtual File System の実体

では、Virtual File System はどのようなデータ構造で実現されているのだろうか。

答えは驚くほど単純である。

実体は

```ts
Map<string, string>;
```

である。

例えば、

```ts
const fsMap = new Map([
  ["/src/index.ts", "import { sum } from './util';"],
  ["/src/util.ts", "export const sum = (a, b) => a + b;"],
]);
```

これだけである。

キーには

```
ファイルパス
```

値には

```
ファイル内容
```

を保存している。

つまり、

```
"/src/index.ts"
            │
            ▼
"import { ... }"
```

という対応表を持っているだけである。

ここで重要なのは、 **Map は「ファイル」を保存しているのではない** という点である。

保存しているのは、単なる

```
キーと値
```

である。

JavaScript の `Map` は、「これはファイルです」という意味を持っていない。

したがって、この段階では Virtual File System は完成していない。

---

### コラム：Map はファイルシステムではない

初心者が最も誤解しやすい点がここである。

例えば、

```ts
fsMap.get("/src/index.ts");
```

と書けば、ファイルの内容は取得できる。

しかし、TypeScript Compiler は

```ts
fsMap.get(...)
```

を呼ぶことはできない。

Compiler が知っているのは、

```
readFile()

fileExists()

directoryExists()
```

といった API だけである。

つまり、Compiler は JavaScript の `Map` を全く知らない。

そのため、Map をそのまま Compiler に渡しても動作しない。

Map を TypeScript Compiler が理解できる形へ変換する必要がある。

その役割を持つのが、次章で詳しく見る **`createSystem()`** である。

---

## この節のまとめ

本節では、Virtual File System の基本概念を説明した。

重要な点は次のとおりである。

- Virtual File System は OS を再現するものではない。
- TypeScript Compiler が必要とする最小限のファイルシステムを提供する。
- 実体は `Map<string, string>` である。
- ただし、`Map` 単体では Compiler は利用できない。
- `Map` を Compiler が理解できる形へ変換する処理が必要になる。
