# 第1章 はじめに

## 1.1 `@typescript/vfs` とは

`@typescript/vfs` は、TypeScript Compiler API を**実際のファイルシステムを使わず**に利用するためのライブラリである。

VFS は **Virtual File System（仮想ファイルシステム）** の略であり、ブラウザやメモリ上のデータを「ファイル」として TypeScript Compiler に認識させる役割を持つ。

通常、TypeScript Compiler は OS 上のファイルを読みながら処理を行う。

例えば次のようなプロジェクトがあるとする。

```text
project/
├── src/
│   ├── index.ts
│   └── util.ts
├── tsconfig.json
└── package.json
```

Compiler は、

1. `tsconfig.json`
2. `index.ts`
3. `util.ts`
4. `lib.dom.d.ts`
5. `lib.es2023.d.ts`

などを順番に読み込み、型解析や JavaScript の生成を行う。

つまり Compiler は、

> 「ファイルが存在する」

ことを前提として設計されている。

---

ブラウザには、このようなファイルシステムは存在しない。

そのため `@typescript/vfs` は、

```ts
const fsMap = new Map([
  ["/index.ts", "const x = 1;"],
  ["/util.ts", "export const y = 2;"],
]);
```

のような `Map<string, string>` を利用し、

> **「この Map がファイルシステムである」**

と TypeScript Compiler に見せる。

Compiler から見ると、

```text
/index.ts
```

というファイルが実際に存在しているように扱われる。

その結果、ブラウザ上でも TypeScript Compiler API を利用できる。

---

## 1.2 なぜ必要なのか

TypeScript Compiler は非常に柔軟な API を持っているが、

基本的には

- Node.js
- Deno

など、

**ファイルシステムが存在する環境**

を前提としている。

例えば Compiler は、

```ts
ts.sys.readFile(...)
```

のような API を利用してソースコードを取得する。

しかしブラウザでは、

- `C:\`
- `/Users/...`
- `/home/...`

のようなディレクトリは存在しない。

つまり、

```text
Compiler
    ↓
ファイルを読む
```

という処理が成立しない。

そこで、

```text
Compiler
    ↓
Virtual File System
    ↓
Map
```

という構成を作る。

Compiler は「ファイルを読んでいる」と思っているが、

実際には `Map` から文字列を取得しているだけである。

このように、

**Compiler を変更せず、ファイルシステムだけを置き換える**

ことが `@typescript/vfs` の目的である。

---

## 1.3 TypeScript Compiler API との関係

ここで誤解しやすい点がある。

`@typescript/vfs` は、

TypeScript Compiler を実装したライブラリではない。

Compiler 自体は、

すべて `typescript` パッケージが提供している。

つまり、

```text
typescript
```

が本体であり、

```text
@typescript/vfs
```

は補助ライブラリである。

役割を整理すると、

| ライブラリ | 役割 |
|------------|------|
| `typescript` | Compiler、Parser、Language Service を提供する |
| `@typescript/vfs` | Compiler が利用する仮想ファイルシステムを構築する |

そのため、

型解析

```ts
languageService.getSemanticDiagnostics(...)
```

コード補完

```ts
languageService.getCompletionsAtPosition(...)
```

JavaScript の生成

```ts
program.emit()
```

などは、

すべて TypeScript 本体が行っている。

`@typescript/vfs` は、

それらを動かすための環境を整えているだけである。

---

## 1.4 設計思想

ソースコード全体を見ると、

`@typescript/vfs` の設計思想は非常に一貫している。

それは、

> **TypeScript の API を隠さない**

ということである。

例えば、

`createVirtualTypeScriptEnvironment()` は、

独自の Language Service を返さない。

返ってくるのは、

```ts
ts.LanguageService
```

そのものである。

つまり、

利用者は通常の Compiler API をそのまま利用できる。

`@typescript/vfs` が提供するものは、

主に次の初期化処理である。

- Virtual File System
- System
- CompilerHost
- LanguageServiceHost

これらを組み立てることで、

TypeScript Compiler が動作できる環境を構築している。

つまり、

```text
TypeScript Compiler
        ▲
        │
@typescript/vfs
```

という関係であり、

`@typescript/vfs` は Compiler を置き換えるライブラリではなく、

Compiler を利用しやすくするライブラリとして設計されている。

## この章のまとめ

本章では `@typescript/vfs` の役割を説明した。

重要なのは次の3点である。

- `@typescript/vfs` は Compiler ではない。
- 仮想化しているのはファイルシステムだけである。
- TypeScript Compiler API をブラウザでも利用できるようにするための補助ライブラリである。
