# 第6章 API リファレンス（前編 1/3）

第1章から第5章までで、`@typescript/vfs` がどのような構造を持ち、Virtual File System を利用して TypeScript Compiler API をブラウザ上でも利用できるようにしているのかを説明した。

本章では、ライブラリが公開している主要 API を一つずつ整理する。

目的は、「どの API が何をするのか」 だけではない。

それぞれの API が、TypeScript Compiler API のどの機能と対応しているのか、また、どのような場面で利用されることを想定しているのかも合わせて理解する。

なお、本章はリファレンスであるため、実装の詳細よりも、利用者から見た役割を中心に説明する。

---

# 6.1 API の全体像

`@typescript/vfs` が公開している API は、大きく分けると 次の四つの分類になる。

```text
Virtual File System
        │
        ├─ Map の生成
        │
        ├─ System の生成
        │
        ├─ CompilerHost の生成
        │
        └─ Environment の生成
```

つまり、ライブラリ全体は 一つの巨大な API ではなく、複数の小さな Factory Function を組み合わせる設計となっている。

---

# 6.2 createDefaultMapFromNodeModules()

## 概要

`createDefaultMapFromNodeModules()` は、Virtual File System を構築するための 初期 Map を生成する API である。

TypeScript が動作するためには、利用者が作成したソースコードだけでは足りない。

例えば、

```ts
const value: string = "hello";
```

というコードであっても、TypeScript は

```
lib.es5.d.ts

lib.dom.d.ts

...
```

などの標準ライブラリ定義ファイルを必要とする。

この API は、それらを含んだ初期状態の Map を構築する。

---

## 入力

概念的には、次の情報を受け取る。

- TypeScript 本体
- CompilerOptions
- 利用する標準ライブラリ

これらを利用して、必要なライブラリファイルを Virtual File System へ登録する。

---

## 出力

戻り値は、

```
Map<string, string>
```

である。

キーにはファイルパス、値にはファイル内容が格納される。

例えば、概念的には

```text
Map
├─ /lib.d.ts
├─ /lib.es5.d.ts
├─ /lib.dom.d.ts
└─ ...
```

のような状態になる。

---

## 利用場面

もっとも一般的なのは、Playground の初期化である。

```text
Map を作る
↓
サンプルコードを追加
↓
System を生成
↓
Environment を生成
```

という流れの、最初の一歩となる。

---

## TypeScript Compiler API との関係

この API 自体は、Compiler を生成しない。

役割は、Compiler が利用する 標準ライブラリを準備することである。

言い換えると、Compiler が期待する 「ファイルシステムの初期状態」 を構築している。

---

# 6.3 createSystem()

## 概要

`createSystem()` は、Virtual File System を TypeScript Compiler API が利用できる `ts.System` へ変換する。

これは、第3章で説明した Adapter に相当する。

---

## 入力

入力は、Virtual File System を表す Map である。

```text
Map
↓
createSystem()
↓
System
```

という変換を行う。

---

## 出力

戻り値は、`ts.System` である。

System は、Compiler が必要とする ファイル操作 API を提供する。

例えば、

- readFile()
- writeFile()
- fileExists()
- directoryExists()
- readDirectory()

などである。

---

## 利用場面

System は、CompilerHost を生成する際に利用される。

そのため、通常は 利用者が System を直接操作することは少ない。

Environment 構築の途中で利用される 内部的な部品と言える。

---

## TypeScript Compiler API との関係

TypeScript Compiler は、Node.js の `fs` モジュールへ直接アクセスしない。

必ず `ts.System` を経由する。

そのため、Virtual File System を利用する場合は、Map を System へ変換する必要がある。

---

# 6.4 createVirtualCompilerHost()

## 概要

`createVirtualCompilerHost()` は、Virtual File System 上で利用できる CompilerHost を生成する。

CompilerHost は、Compiler が ソースファイルを取得するための 情報提供インターフェースである。

---

## 入力

概念的には、次の情報を受け取る。

- System
- CompilerOptions
- TypeScript

これらを利用して、Virtual File System を利用する CompilerHost を構築する。

---

## 出力

戻り値は、CompilerHost である。

この Host を利用することで、Compiler は 実ファイルシステムではなく、Virtual File System 上のソースコードを解析できる。

---

## 利用場面

通常、利用者は この API を直接利用することは少ない。

多くの場合は、`createVirtualTypeScriptEnvironment()` の内部から利用される。

しかし、Compiler API を直接操作したい場合には、この API を単独で利用することもできる。

---

## TypeScript Compiler API との関係

CompilerHost は、TypeScript Compiler API の中心的なインターフェースの一つである。

VFS は、CompilerHost を Virtual File System 向けに実装することで、ブラウザ上でも 通常の Compiler を利用できるようにしている。

---

## この節で理解できるようになったこと

- API は役割ごとに小さく分割されている。
- `createDefaultMapFromNodeModules()` は初期 Map を生成する。
- `createSystem()` は Map を `ts.System` へ変換する。
- `createVirtualCompilerHost()` は Virtual File System 用の CompilerHost を生成する。
- それぞれは独立した Factory Function として設計されている。

---

## 次回（第6章 前編 2/3）

次回は、ライブラリの中心 API である `createVirtualTypeScriptEnvironment()` を中心に、Environment の生成手順と公開メソッドを整理する。

ここまで学んできた

- Map
- System
- CompilerHost

が、どのように一つの実行環境へ統合されるのかを見ていく。

# 第6章 API リファレンス（前編 2/3）

前節では、Virtual File System を構築するための基本 API を紹介した。

ここまでの流れを整理すると、

```text
Map
↓
System
↓
CompilerHost
```

という順番で、TypeScript Compiler API が利用できる環境を準備してきた。

しかし、実際に TypeScript Playground のような環境を構築するには、まだ十分ではない。

- コード補完
- 型チェック
- 定義ジャンプ

などの機能を利用するには、Language Service を生成し、それらを一つのオブジェクトとして扱う必要がある。

その役割を担うのが、`createVirtualTypeScriptEnvironment()` である。

---

# 6.5 createVirtualTypeScriptEnvironment()

## 概要

`createVirtualTypeScriptEnvironment()` は、`@typescript/vfs` を利用する際の中心となる API である。

利用者は、通常この API を呼び出すことで、Virtual File System 上で動作する TypeScript 開発環境を取得する。

概念図で表すと、

```text
Virtual File System
        │
        ▼
System
        │
        ▼
CompilerHost
        │
        ▼
LanguageServiceHost
        │
        ▼
LanguageService
        │
        ▼
VirtualTypeScriptEnvironment
```

これまで個別に生成してきた部品を、一つの Environment としてまとめる Factory Function である。

---

## 入力

概念的には、次のような情報を受け取る。

| 引数            | 役割                           |
| --------------- | ------------------------------ |
| System          | Virtual File System を提供する |
| Root Files      | プロジェクトの開始ファイル     |
| TypeScript      | Compiler API 本体              |
| CompilerOptions | コンパイル設定                 |

これらを利用して、Language Service が利用できる状態を構築する。

---

## 出力

戻り値は、`VirtualTypeScriptEnvironment` である。

これは、Language Service 単体ではなく、TypeScript の実行環境全体を表すオブジェクトである。

利用者は、この Environment を通して、ファイル操作や型解析を行う。

---

## 利用場面

もっとも一般的な利用例は、ブラウザ IDE の初期化である。

概念的には、次のような流れになる。

```text
Map を生成
↓
サンプルコードを追加
↓
System を生成
↓
Environment を生成
↓
コード補完・型チェック開始
```

Environment が生成されると、以後は Environment を中心に操作を行う。

---

# 6.6 VirtualTypeScriptEnvironment の主要プロパティ

Environment は、Language Service を隠蔽するだけのオブジェクトではない。

内部では、複数の重要なオブジェクトを保持している。

代表的なものは、

```text
Environment
├─ LanguageService
├─ LanguageServiceHost
├─ CompilerHost
├─ System
└─ Virtual File System
```

である。

利用者は、通常これらを直接操作する必要はない。

Environment が、必要な順序で内部オブジェクトを利用する。

---

# 6.7 updateFile()

## 概要

`updateFile()` は、既存ファイルの内容を更新するための API である。

ブラウザエディタでは、キー入力のたびに この API が利用されることになる。

---

## 入力

概念的には、次の情報を受け取る。

| 引数               | 内容         |
| ------------------ | ------------ |
| ファイル名         | 更新対象     |
| 新しいソースコード | 更新後の内容 |

---

## 処理

Environment は、単純に Map を更新するわけではない。

内部では、概念的に次の処理を行う。

```text
ファイル内容更新
↓
Version 更新
↓
Language Service が変更を認識
↓
必要な解析を再実行
```

ここで重要なのは、Language Service が 変更を認識できる状態を維持することである。

---

## 利用場面

例えば、エディタで

```ts
const value = 100;
```

を

```ts
const value = "hello";
```

へ変更した場合、Environment は 更新後のソースを保持し、Language Service は 新しい型情報を用いて エラーや補完候補を再計算する。

---

# 6.8 createFile()

## 概要

`createFile()` は、Virtual File System に 新しいファイルを追加する。

例えば、

```
index.ts
```

しか存在しないプロジェクトへ、

```
math.ts
```

を追加するような場面で利用される。

---

## 処理

概念的には、

```text
ファイル追加
↓
Version 初期化
↓
Project 更新
↓
Language Service へ反映
```

という流れになる。

---

## 利用場面

オンライン IDE で、「新規ファイル」 ボタンを押した場合などが代表例である。

Environment を利用することで、利用者は Language Service の更新処理を意識する必要がない。

---

# 6.9 deleteFile()

## 概要

`deleteFile()` は、Virtual File System から ファイルを削除する。

---

## 処理

Environment は、概念的に 次のような処理を行う。

```text
Virtual File System
↓
ファイル削除
↓
Project 更新
↓
Language Service 更新
```

---

## 利用場面

例えば、不要になった

```
sample.ts
```

を削除した場合、Language Service は そのファイルを 解析対象から除外する。

これにより、存在しないファイルに対する エラーや補完候補が残り続けることを防いでいる。

---

## この節で理解できるようになったこと

- `createVirtualTypeScriptEnvironment()` はライブラリの中心 API である。
- Environment は TypeScript の実行環境全体を表す。
- `updateFile()` はファイル更新と状態同期を行う。
- `createFile()` は新規ファイルをプロジェクトへ追加する。
- `deleteFile()` は削除後の状態まで管理する。

# 6.10 emit()

## 概要

`emit()` は、TypeScript のソースコードから JavaScript を生成するための API である。

TypeScript Compiler API では、この処理を **Emit** と呼ぶ。

VirtualTypeScriptEnvironment は、Compiler API が持つ emit 機能を、利用しやすい形で公開している。

---

## 入力

一般的には、対象となる SourceFile、またはプロジェクト全体を対象として実行される。

CompilerOptions によって、生成される JavaScript の形式は変化する。

例えば、

- 出力先
- ECMAScript のバージョン
- Source Map の生成
- Declaration File の生成

などは、

CompilerOptions の設定に従う。

---

## 処理

概念図で表すと、

```text
SourceFile
        │
        ▼
Compiler
        │
        ▼
JavaScript
        │
        ▼

出力結果
```

Environment 自身が JavaScript を生成しているわけではない。

実際の変換処理は、TypeScript Compiler が担当する。

Environment は、Compiler を適切な状態で呼び出す役割を持つ。

---

## 利用場面

代表例として、Playground の

```
JavaScript 出力
```

タブがある。

ユーザーが TypeScript を編集すると、Environment が最新状態を保持し、emit が実行され、最新の JavaScript が生成される。

---

# 6.11 LanguageService へのアクセス

VirtualTypeScriptEnvironment は、Language Service を完全に隠蔽するライブラリではない。

高度な機能を利用したい場合は、Environment が保持している Language Service を利用することもできる。

例えば、Language Service が持つ代表的な機能として、

- コード補完
- 型情報取得
- 定義ジャンプ
- 参照検索
- リネーム
- Quick Info
- Signature Help

などがある。

Environment は、これらを実現するための 実行環境を管理している。

つまり、Environment は Language Service の代替ではなく、Language Service を利用しやすくするラッパーと言える。

---

# 6.12 API の責務

ここまで紹介してきた API を整理すると、次のようになる。

| API                                  | 主な役割                 |
| ------------------------------------ | ------------------------ |
| createDefaultMapFromNodeModules()    | 初期ファイル群を生成する |
| createSystem()                       | Map を System へ変換する |
| createVirtualCompilerHost()          | CompilerHost を生成する  |
| createVirtualTypeScriptEnvironment() | Environment を生成する   |
| updateFile()                         | ファイル内容を更新する   |
| createFile()                         | 新規ファイルを追加する   |
| deleteFile()                         | ファイルを削除する       |
| emit()                               | JavaScript を生成する    |

これらは、それぞれ独立した役割だけを持っている。

一つの API が 複数の責務を持たないことが、ライブラリ全体の特徴である。

# 6.13 TypeScript Compiler API との対応

`@typescript/vfs` は、TypeScript Compiler API を 置き換えるライブラリではない。

むしろ、Compiler API を利用しやすくするための 補助ライブラリである。

概念図で表すと、

```text
@typescript/vfs
        │
        ├─ Virtual File System
        ├─ System
        ├─ CompilerHost
        └─ Environment
                │
                ▼
TypeScript Compiler API
        │
        ├─ Compiler
        ├─ Program
        ├─ Language Service
        └─ Emit
```

つまり、VFS は Compiler API の上位レイヤーとして設計されている。

---

# 6.14 API 設計の特徴

最後に、`@typescript/vfs` 全体に共通する設計思想を整理する。

## Factory Function を中心とする

公開 API の多くは、`create` から始まる。

これは、TypeScript Compiler API と 同じ設計思想である。

オブジェクト生成を Factory Function に集約することで、初期化処理を利用者から隠蔽している。

---

## 小さな部品を組み合わせる

ライブラリは、巨大なクラスではなく、小さな部品を組み合わせる設計になっている。

例えば、

```text
Map
↓
System
↓
CompilerHost
↓
Environment
```

という流れは、一方向であり、各レイヤーが 一つの責務だけを持つ。

この構造は、テストしやすく、拡張もしやすい。

---

## TypeScript 本体へ処理を委譲する

VFS は、型解析や コード補完を 再実装していない。

実際の処理は、全て TypeScript Compiler API へ委譲している。

そのため、VFS は 比較的小規模なライブラリでありながら、TypeScript 本体の強力な機能を利用できる。

---

# 6.15 この章のまとめ

本章では、`@typescript/vfs` が公開している 主要 API を整理した。

各 API は、

- Virtual File System の構築
- CompilerHost の生成
- Environment の生成
- ファイル更新
- JavaScript 出力

など、一つの責務だけを担当している。

また、ライブラリ全体を通して、Factory Function を中心とした設計、責務の分離、そして TypeScript Compiler API への委譲 という思想が一貫して採用されている。

これにより、利用者は 複雑な Compiler API を直接扱うことなく、ブラウザ上で TypeScript 開発環境を構築できる。

---

## この章で理解できるようになったこと

- `emit()` は TypeScript Compiler の出力処理を利用する。
- Environment は必要に応じて Language Service へアクセスする窓口となる。
- 公開 API はそれぞれ単一の責務を持つ。
- `@typescript/vfs` は Compiler API の上位レイヤーとして設計されている。
- ライブラリ全体は Factory Function を中心とした構成になっている。
