# 第6章 `createVirtualTypeScriptEnvironment()` を読む

ここまでで、`@typescript/vfs` を構成する土台となる二つのレイヤーを見てきた。

```
Map
    ↓
System
```

そして、

```
System
    ↓
CompilerHost
```

である。

しかし、これだけではまだ利用者が普段使う API には到達していない。

実際にブラウザ上で TypeScript Playground のような環境を構築するとき、多くの利用者が最初に呼び出すのは

```ts
createVirtualTypeScriptEnvironment(...)
```

である。

本章では、このライブラリの中心とも言える関数を読み解いていく。

---

# 6.1 この関数は何を解決するのか

まず、ソースコードを見る前に考えてみよう。

ここまで作ってきたものは、

```
Map

System

CompilerHost
```

だけである。

これは言い換えると、

> Compiler が動くための部品

を一つずつ組み立ててきたに過ぎない。

まだ

```
getSemanticDiagnostics()

getCompletionsAtPosition()

findReferences()

emit()
```

など、

エディタが必要とする機能は利用できない。

これらはすべて

```
Language Service
```

という別の仕組みが担当している。

つまり、

利用者が本当に欲しいものは

```
CompilerHost
```

ではなく、

```
LanguageService
```

なのである。

---

ここで

`createVirtualTypeScriptEnvironment()`

の役割が見えてくる。

この関数は、

これまで組み立ててきた部品を利用して、

最終的に

```
LanguageService
```

を利用できる状態まで初期化する。

つまり、

責務を一文で表すと、

> **Virtual File System 上で動作する TypeScript 開発環境を構築すること**

である。

この「開発環境」という言葉が、

Environment という名前の由来になっている。

---

## 6.2 全体の流れ

この関数が内部で行う処理を、

細かな実装を省略して概念だけで表すと、

次のようになる。

```text
Map
 │
 ▼
createSystem()

 │
 ▼
createVirtualCompilerHost()

 │
 ▼
LanguageServiceHost を生成

 │
 ▼
ts.createLanguageService()

 │
 ▼
VirtualTypeScriptEnvironment を返す
```

重要なのは、

この関数自身が

型解析

コード補完

リファクタリング

などを実装しているわけではない

ということである。

それらは

すべて

TypeScript Compiler API

が提供している。

`@typescript/vfs` は、

その API を利用できるように、

必要な部品を順番に組み立てているだけである。

---

## 6.3 関数シグネチャを見る

まず、

実装の先頭を見る。

概念的には次のような形になっている。

```ts
export function createVirtualTypeScriptEnvironment(...)
```

ここで注目したいのは、

実装の中身ではなく、

シグネチャそのものから読み取れる情報である。

---

### `export`

この関数は、

ライブラリ利用者へ公開されている API である。

つまり、

作者は

> 利用者が直接呼び出す入口

として設計している。

逆に、

内部専用の補助関数ではないことが分かる。

---

### `function`

クラスではなく、

Factory Function が採用されている。

これは第2章でも触れたように、

`@typescript/vfs`

全体で共通する設計思想である。

状態を持つクラスを公開するのではなく、

必要なオブジェクトを組み立てて返す。

これにより、

利用者は

```ts
new Environment(...)
```

のような生成手順を意識する必要がない。

---

### 関数名

関数名を分解すると、

```
create

Virtual

TypeScript

Environment
```

となる。

それぞれ、

- create
    - 新しい環境を構築する

- Virtual
    - 実ファイルシステムではなく VFS を利用する

- TypeScript
    - JavaScript 全般ではなく TypeScript Compiler 用

- Environment
    - Compiler 単体ではなく、開発環境全体を意味する

という意図が読み取れる。

このライブラリでは、

関数名が非常に説明的に付けられている。

---

## 6.4 なぜ "Environment" なのか

ここで一つ疑問がある。

なぜ

```
createLanguageService()
```

ではなく、

```
createVirtualTypeScriptEnvironment()
```

なのだろうか。

これは、

戻り値が

Language Service

だけではないからである。

実際には、

Environment は

Language Service に加えて、

- 仮想ファイルの更新
- 仮想ファイルの追加
- 仮想ファイルの削除
- CompilerHost
- 各種状態管理

など、

エディタが必要とする機能をまとめて保持している。

つまり、

Language Service を中心とした

**実行環境**

全体を返しているため、

Environment

という名前が選ばれている。

この命名は、

ライブラリの責務を非常によく表している。

---

## コラム：Playground が欲しいのは Compiler ではない

TypeScript Playground や Web IDE を作る場合、

本当に欲しいのは

Compiler

ではない。

欲しいのは、

```
入力する

↓

補完される

↓

型エラーが出る

↓

定義へジャンプできる
```

という一連の体験である。

これらは

Compiler

だけでは実現できない。

必要なのは

Language Service

である。

`createVirtualTypeScriptEnvironment()`

は、

Compiler を作るための API ではなく、

Language Service を利用できる環境を作る API

なのである。

この点を理解すると、

この関数がライブラリの中心 API である理由も自然に理解できる。

---

## この章で理解できるようになったこと

- `createVirtualTypeScriptEnvironment()` はライブラリの入口となる公開 API である。
- この関数自身は型解析を実装しているわけではない。
- 内部で各種オブジェクトを組み立て、Language Service を利用できる状態を作る。
- "Environment" という名前は、Language Service を含む開発環境全体を返すことを表している。
